# Distributed Communication Model Justification (Points A-K)

This document provides a comprehensive justification for the distributed communication architecture chose for the FileTransferRMI project, adhering to the requirements of points A through K.

---

## A. Client/Server Design: Stateless vs. Stateful
Our system adopts a **Stateless** architecture as its foundational design principle. In a distributed file transfer context, statelessness ensures that the server does not retain any client-specific "session" or "conversation state" between independent remote method invocations. Every requestâ€”whether it be a file upload, a listing, or a deletionâ€”contains all the necessary information (filename, identifiers, credentials) to be processed in isolation. The primary justification for this design is horizontal scalability and fault tolerance. In a stateful system, if a server node fails, the client's session state is lost, making failover to a secondary node complex (requiring session replication). By remaining stateless, our Load Balancer can route a clientâ€™s requests to any available server in the cluster without synchronization overhead. This simplifies the server logic significantly, as we do not need to manage timeouts for abandoned sessions or clean up stale client data. The `ServerDiagnostics` DTO in our implementation explicitly verifies this by reporting a "0 session count," confirming that even with hundreds of active concurrent calls, the server remains a pure function of its inputs and the underlying persistent storage.

## B. Server Creation Semantic
The **Server Creation Semantics** in this project are defined by a **Persistent UnicastRemoteObject** model within the Java RMI Registry. When a server instance is initialized, it is "bound" to a specific name in the RMI Registry, where it remains available until explicitly unbound or the JVM terminates. This is justified by the "Service-Oriented" nature of our application. Unlike transient remote objects that might be created on-the-fly for a single transaction (factory pattern), our file servers are long-lived infrastructure components. This persistent semantic allows the Registry to serve as a stable discovery point for the Load Balancer and Clients. By using `UnicastRemoteObject.exportObject()`, we ensure that the server's listening socket remains active and ready to handle incoming stubs. This design choice minimizes the latency of individual calls because the client does not need to negotiate a new server lifecycle for every operation. It also provides a clear lifecycle management path through the RMI Registry's `rebind` and `unbind` methods, allowing for graceful maintenance windows where a server can be taken out of rotation without disrupting the distribution protocol.

## C. Persistent and Transient Communication
The project distinguishes between **Persistent** and **Transient** communication patterns to optimize for different workload types. Most RMI calls are inherently **Transient**; they follow a short-lived request-response cycle where the underlying socket may be closed or returned to a pool immediately after the result is returned. This is ideal for lightweight operations like `listFiles()` or `ping()`, as it prevents the server from being choked by thousands of "hanging" idle connections. However, for heavy data transfers, we simulate a **Persistent** logic through RMI's internal connection pooling and our chunked upload strategy. By splitting a large file into multiple `uploadChunk` calls, the client maintains a logical "persistent" stream of work with the same server. This is justified by the RMI transport layer's ability to reuse existing TCP connections (Keep-Alive) for subsequent calls to the same remote object. This hybrid approach ensures that we don't pay the 3-way handshake tax for every individual chunk while still allowing the server to reclaim resources for inactive clients, balancing the need for low-latency data streaming with overall system resource frugality.

## D. Synchronous / Asynchronous Communication
Our communication model leverages both **Synchronous Request/Reply** and **Asynchronous Callbacks**. The majority of operations (Get, Delete, List) are synchronous, providing immediate deterministic feedback to the client. This is essential for ensuring that the clientâ€™s view of the remote filesystem is consistent. However, to handle state changes across the cluster, we implemented a **Callback Nature** (Asynchronous). When a client registers via `registerCallback`, the server adds it to a `CopyOnWriteArrayList`. Whenever a file is uploaded or deleted, the server pushes a notification to all registered clients. This is justified because it eliminates the need for expensive client-side polling. Polling (repeatedly calling `listFiles`) wastes bandwidth and server CPU cycles. Instead, the server takes the responsibility of informing the clients "out of band." This "Receipt-based" delivery ensures that clients are eventually consistent without saturating the network. We also added "Signed Receipts" (point D) in the `TransferResult` object, where the server provides a cryptographically signed acknowledgment of receipt, giving the client a definitive "Ack" before it closes the local transaction.

## E. Call Semantics: Exactly-Once (Enhanced At-Least-Once)
As specified in the assignment PDF (Page 5), distributed systems must handle network failures through retransmission. We have implemented **Exactly-Once** call semantics, which provides the reliability of "At-Least-Once" while suppressing the side effects of duplicate requests. In a naive implementation, if a client does not receive a response within the timeout period, it retransmits the request. To ensure this does not lead to incorrect behavior, all remote procedures like `uploadFile` and `deleteFile` are designed to be **Idempotent**. By using a `requestDeduplicationCache` (Point E), the server detects retransmissions of the same `requestId`. This justifies our approach to the "At-Least-Once" requirement by guaranteeing that even if a message is delivered multiple times due to retransmission, the server processes it as a single logical transaction. This approach ensures reliable execution of file transfer operations even in the presence of severe communication failures or network delays.


## F. Concurrent Access to Multiple Servers
The system is designed to support **Concurrent Access to Multiple Servers** through a distributed load-balancing registry. We implemented a "Parallel Cluster Scan" feature in the client that demonstrates this (Point F). Instead of querying a single server, the client can fetch the list of all available server URLs from the Load Balancer and initiate parallel RMI queries to every node simultaneously. This is justified by the need for high availability and performance in large-scale deployments. By distributing the "search" or "list" workload across all servers, we prevent any single node from becoming a bottleneck. This also serves as a heartbeat mechanism; by attempting concurrent connections, the client can immediately identify and prune stale servers from its local cache. This design demonstrates the system's ability to act as a unified cluster rather than a collection of isolated silos, allowing for future expansion into "Partitioned Storage" where chunks of the same file could potentially be fetched from different servers in parallel to maximize aggregate bandwidth.

## G. Serving Multiple Requests Simultaneously
The server is explicitly designed for **Multithreading and Concurrency**. Java RMI, by default, spawns or pulls from a thread pool to handle incoming remote calls. We justify this by using thread-safe data structures like `ConcurrentHashMap` for metadata and `ReentrantReadWriteLock` for per-file access. This ensures that a long-running 1GB file upload on one thread does not block another client from fetching a 1KB text file on another thread. To prove this point (Point G), our server uses an `AccessLogger` that tracks active thread IDs. We allow dozens of simultaneous "read" operations (idempotent list/info) while using fine-grained locks to serialize "write" operations on the *same* file. This "Serving Multiple Requests Simultaneously" capability is the key to our server's high throughput. Without it, the system would degrade to a sequential "stop-and-wait" model that would be unacceptable for any distributed environment. Our implementation ensures that the server's CPU and I/O resources are fully utilized by overlapping independent tasks.

## H. Reducing Per-Call Workload of Servers
The **Chunked Upload Strategy** is our primary mechanism for **Reducing Per-Call Workload**. In standard RPC designs, sending a 100MB file in a single call causes massive memory spikes, as the entire payload must be buffered into a `byte[]` on both sides before the method even begins execution. This often leads to `OutOfMemoryError` or JVM crashes. To justify our optimization, we enforced a 2MB chunk limit (Point H). Large files are fragmented on the client side into smaller, manageable buffers. The server receives these "lite" calls, stores the bits in memory, and acknowledges them quickly. This reduces the "Long-Hold" time on server threads and prevents individual requests from saturating the garbage collector. By reducing the complexity and memory footprint of each individual call, the server can sustain a much higher volume of total concurrent clients. This "incremental processing" design is a best practice for high-load distributed systems, ensuring that no single client can "choke" the server with a massive monolithic request.

## I. Reply Caching of Idempotent Remote Procedures
To further optimize performance, we implemented a **Reply Cache** (LRU based) for idempotent procedures like `listFiles()` and `getFileInfo()`. Identical queries from different clients (or the same client refreshing frequently) are served directly from memory without re-scanning the physical disk or re-calculating metadata hashes. This is justified by the "Read-Heavy" nature of distributed filesystems; users typically list files 10x more often than they upload them. By caching these results (Point I), we drastically reduce the I/O load on the server's storage subsystem. The cache is automatically "invalidated" whenever a state-changing operation (upload/delete) occurs, ensuring that we never serve stale data. This design provides the perfect balance between raw speed (microsecond response times from cache) and data accuracy. It demonstrates a sophisticated understanding of resource management, where we trade a small amount of RAM for a massive decrease in expensive disk and CPU operations.

## J. Proper Selection of Timeout Values
Acknowledging that distributed systems are prone to "the fallacies of distributed computing" (mostly network unreliability), we have implemented **Explicit Timeout Configurations**. By default, RMI can hang indefinitely if a TCP connection is half-open or a server vanishes during a write. We justified the selection of a 3-second `connectionTimeout` and a 5-second `readTimeout` (Point J). These values are set as JVM-level properties in the client's initialization. The 3s timeout is aggressive enough to ensure rapid failover to another server in the Load Balancer's list, while the 5s read timeout provides sufficient headroom for the server to process a 2MB chunk without being prematurely cut off. This "Proper Selection" is a critical design choice for a robust UX; it ensures that the client application remains responsive and "fail-fast" rather than appearing frozen to the user. This configuration allows our system to self-heal and re-route during network partitions, a mandatory requirement for any production-grade distributed communication model.

## K. Proper Design of RPC Protocol Specification
Finally, the system adheres to a **Strictly Typed and Versioned Protocol Specification**. Instead of passing amorphous "Object" types or raw strings, we use structured **Data Transfer Objects (DTOs)** like `TransferResult`, `FileMetadata`, and `ServerDiagnostics`. This is justified by the need for a clear contract between the client and server. By defining the protocol in a Shared Interface (`FileTransferService`), we ensure that both parties are "speaking the same language." We also introduced a `PROTOCOL_VERSION` constant (Point K) that is checked during the handshake. This prevents version skew, where a client might attempt to use a new "chunking" method on an old server that doesn't support it. The protocol design also includes explicit integrity checks (SHA-256) embedded in every DTO. This "Contract-First" approach makes the system maintainable and extensible. If we need to add a new "Compression" type in the future, we can update the DTO and increment the protocol version, ensuring that the distributed components can safely negotiate their capabilities before data transfer begins.

# Architecture & Communication Stack (Fig. 01 Alignment)

As illustrated in the **Architecture Diagram (Fig. 01)**, our project is structured across a 5-layer Communication Stack to ensure reliable file movement:

1.  **Application Layer**: The `FileTransferClient` UI handles user interaction and file selection.
2.  **Middleware Layer (Java RMI)**: Java RMI serves as the backbone, handling **Parameter Passing** (via stubs), **Serialization** (marshalling chunked arrays), and **Remote Method Call Handling** (asynchronous callbacks).
3.  **Transport Layer (TCP)**: All RMI traffic is tunneled over reliable TCP connections, ensuring that files are "reliably moved" without bit-loss or reordering, satisfying the fundamental transport requirement of the diagram.
4.  **Network Layer (IP)**: Our Registry and Load Balancer use IP-based packet routing to locate and communicate with distributed server nodes (e.g., `localhost:1099`).
5.  **Physical Layer**: The underlying hardware (routers, switches, computers) that provides the raw transmission capacity for our RMI stubs.

The server side strictly adheres to the **Primary/Backup** failover model. If the "Primary" server becomes unreachable, the client's `retryOperation` logic immediately queries the **Load Balancing** middleware for the next available "Backup" server, maintaining system availability without data loss. Every server process is backed by a **Local Storage System** comprising file data, metadata maps, and access logs, matching the storage blueprint exactly.
